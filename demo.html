Complete Documentation

1 Feature Overview

The Spend Limit feature allows Organizations (e.g., Fonespot or Admins) to allocate specific spending balances to individual stores. Stores can then use this "Spend Limit" for various transactions (like stock requests or repairs). The system ensures that total allocations do not exceed the organization's overall wallet capacity.

Key Benefit: Centralized control over store spending with real-time balance tracking.

2 Database Schema

pos_organizations Table

Column Type Description
spend_limit decimal The total master balance available for the organization. This acts as the "Wallet".

pos_stores Table

Column Type Description
assigned_limit decimal The maximum limit allocated by the admin to this specific store.
spend_limit decimal The currently available balance for the store (Assigned - Spent).
spent_amount decimal The total amount the store has consumed.

pos_spend_limit_requests Table

Used for stores to request a balance Reset or a limit Increase.

Column Type Description
request_type enum reset / increase
requested_amount decimal Amount requested for increase
status enum pending / approved / rejected

3 Core Logic & Validation

Spend Limit Distribution

Located in WalletController@save

Validation Steps:
  1. Checks if the $amount being assigned is valid.
  2. Wallet Sync: Calculates the difference between the new limit and the old assigned_limit.
  3. Balance Check: Ensures the Organization's spend_limit (Wallet) is greater than the difference.
  4. Error: "Insufficient funds in organization wallet ($X.XX)."
Deduction/Refund Logic:
  • If increasing a store limit, the difference is decremented from the Organization's spend_limit.
  • If decreasing a store limit, the difference is incremented (refunded) to the Organization's spend_limit.
  • Store Reset: When a limit is set or reset, spent_amount is cleared (0) and the spend_limit is set equal to the assigned_limit.
